# SCP-011.1 — Catalog Gate, Strict Input Boundary & Runtime Orchestration Test Hardening

## Status

Accepted

## Objetivo

Corrigir os bloqueios auditados na SCP-011:

1. Restaurar o Commercial Feature Catalog Gate no fluxo de `users.seats`.
2. Tornar o input público estrito (rejeita propriedades adicionais).
3. Testar a orquestração real de produção, não um reader paralelo.
4. Corrigir a evidência documental sobre `limit_reached`.
5. Preservar o caráter read-only integral da SCP-011.

## Confirmação de estado inicial (repositório real)

Inspeção direta antes das mudanças:

- `src/lib/api/commercial/commercial.functions.ts` — `getCommercialSeatLimitDecision`
  chamava `loadAdmin` + `loadTenantCommercialContext` **antes** de qualquer
  catalog gate; `evaluateFeatureCatalogGate` era invocado apenas em
  `getCommercialFeatureDecision`.
- Input validator aceitava `data ?? {}` e ignorava propriedades adicionais
  (`{ requestedIncrement?: unknown }`), sem rejeição estrutural.
- `commercial-seat-limit.spec.ts` cobria apenas os helpers puros
  (`decideCommercialSeatLimit`, `extractSeatLimit`, `normalizeSeatIncrement`,
  `validateSeatUsedCount`) e um `readUsed` local que replicava manualmente
  a query — nenhum teste exercia a orquestração usada em produção.
- `limit_reached` aparecia em:
  - `src/lib/api/commercial/limit-decision.ts:235` (runtime puro emite);
  - `src/lib/api/commercial/limit-decision.ts:200` (comentário);
  - `src/lib/api/commercial/feature-gate.ts:38` (enum);
  - `src/lib/api/commercial/feature-gate.ts:110` (comentário);
  - `src/integrations/supabase/__tests__/commercial-feature-gate.spec.ts:44`;
  - várias linhas em `commercial-seat-limit.spec.ts`.

## Mudanças aplicadas

### 1. Nova orquestração testável (produção == testes)

Arquivo novo: `src/lib/api/commercial/seat-limit-runtime.ts` expõe
`resolveCommercialSeatLimitDecision({ tenantId, requestedIncrement, deps })`
com dependências injetáveis:

```ts
interface SeatLimitRuntimeDeps {
  evaluateCatalogGate: (i) => CommercialFeatureDecision | null;
  loadCommercialContext: (tenantId) => Promise<CommercialContext>;
  readSeatUsage: (tenantId) => Promise<number | null>;
}
```

Ordem canônica (SCP-011.1 §4):

```
requireTenant                             ← server function
→ normalizeCommercialSeatLimitInput       ← server function
→ featureKey = SEAT_FEATURE_KEY           ← server-side
→ deps.evaluateCatalogGate                ← ANTES de qualquer leitura
→ (se catalog OK) deps.loadCommercialContext
→ decideCommercialFeature
→ (se feature OK) extractSeatLimit + deps.readSeatUsage
→ decideCommercialSeatLimit               ← puro, sem I/O
```

### 2. `getCommercialSeatLimitDecision` reescrito

`src/lib/api/commercial/commercial.functions.ts` agora:

- usa `normalizeCommercialSeatLimitInput` como `inputValidator`;
- delega a decisão a `resolveCommercialSeatLimitDecision`, injetando
  `defaultEvaluateCatalogGate` (o mesmo `evaluateFeatureCatalogGate`
  usado por `getCommercialFeatureDecision`), `loadTenantCommercialContext`
  (via `loadAdmin`) e o `COUNT` `tenant_members` (`active`+`invited`);
- não executa nenhuma leitura comercial nem toca `tenant_members`
  quando o catálogo nega a chave (curto-circuito puro).

`getCommercialFeatureDecision` permanece inalterado — o catalog gate
já precedia `loadTenantCommercialContext` (SCP-008); não há regressão.

### 3. Input boundary estrito

`src/lib/api/commercial/limit-decision.ts` ganhou
`normalizeCommercialSeatLimitInput`:

- aceita apenas `undefined`, `{}` e `{ requestedIncrement: 1 }`;
- rejeita `null`, arrays, primitivos, e qualquer propriedade fora do
  whitelist (`{ tenantId, featureKey, used, limit, remaining, source,
  billingStatus, membershipCount, currentSeats, foo, ... }`);
- rejeita `requestedIncrement` diferente de `1` (0, 2, negativos,
  fracionários, strings).

Verificação por `Object.keys(input)` — não há “ignorar silencioso”.

### 4. Testes de orquestração e input

Nova cobertura em `commercial-seat-limit.spec.ts`
(45 testes no total, +11):

- catalog gate negativo → `loadCommercialContext` e `readSeatUsage`
  provados como não chamados (contadores);
- feature negativa em cada razão (`not_entitled`, `billing_unknown`,
  `billing_attention_required`, `billing_blocked`) → `readSeatUsage`
  provado como não chamado; `limit/used/remaining` = null;
- feature positiva → `readSeatUsage` chamado exatamente uma vez, com
  `tenantId` resolvido pelo `requireTenant`; `entitled`/`limit_reached`
  emitidos corretamente; `source` preservado;
- falha de leitura (`readSeatUsage` retorna `null`) → `not_evaluated`,
  jamais `limit_reached`;
- input rejeita propriedades adicionais, `requestedIncrement` != 1,
  arrays, primitivos e `null`.

Os testes importam **o mesmo módulo** consumido pela server function
(`resolveCommercialSeatLimitDecision`) — não há reader paralelo. O
mock chainable existente permanece apenas para o teste estrutural do
formato da query (`tenant_members` + `active/invited` + `count=exact/head`).

### 5. Correção evidencial de `limit_reached`

`rg -n 'limit_reached' src/` executado após a etapa retorna, exatamente:

- `src/lib/api/commercial/limit-decision.ts:200` — comentário “NEVER emit limit_reached”;
- `src/lib/api/commercial/limit-decision.ts:235` — `reason: "limit_reached"` (runtime puro emite);
- `src/lib/api/commercial/feature-gate.ts:38` — membro do enum;
- `src/lib/api/commercial/feature-gate.ts:110` — comentário do enum;
- `src/integrations/supabase/__tests__/commercial-feature-gate.spec.ts:44` — cobertura do enum;
- `src/integrations/supabase/__tests__/commercial-seat-limit.spec.ts` — múltiplas asserções sobre a emissão do valor.

A afirmação anterior (“apenas em enum e specs”) era incompleta: o
runtime puro `decideCommercialSeatLimit` emite `limit_reached`
explicitamente, o que é o comportamento correto e auditado.
