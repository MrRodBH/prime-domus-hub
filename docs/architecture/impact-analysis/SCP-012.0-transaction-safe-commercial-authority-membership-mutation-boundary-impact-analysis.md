# SCP-012.0 — Transaction-Safe Commercial Authority & Membership Mutation Boundary Impact Analysis

## Status

Ready for External Audit

## 1. Objetivo

Executar a análise arquitetural completa dos dois pré-requisitos que impedem
a implementação da SCP-012 (Commercial Seat Limit Atomic Enforcement
Integration):

1. ausência de autoridade comercial executável dentro de uma transação
   Postgres — hoje a decisão comercial existe exclusivamente em TypeScript
   (`src/lib/api/commercial/**`), fora do plano transacional do banco;
2. ausência de um boundary server-side de mutations de membership — hoje
   não existe qualquer server function, RPC, repository, hook ou fluxo de
   invitation/aceite que altere `public.tenant_members` em runtime.

Esta etapa é exclusivamente de planejamento, documentação e confirmação de
estado. Nenhuma migration, RPC, mutation ou enforcement é criado.

## 2. Estado atual confirmado

  SCP-011.3.3 = **Accepted**.

## 4. Schema, RLS e grants (DDL real)

### 4.1 Tabela `public.tenant_members`

Coalesced a partir de `20260701204508` + `20260708125042`:

```sql
CREATE TABLE public.tenant_members (
  tenant_id          uuid    NOT NULL REFERENCES public.tenants(id) ON DELETE CASCADE,
  user_id            uuid    NOT NULL REFERENCES auth.users(id)      ON DELETE CASCADE,
  is_owner           boolean NOT NULL DEFAULT false,
  is_default         boolean NOT NULL DEFAULT true,
  joined_at          timestamptz NOT NULL DEFAULT now(),
  tenant_role        public.tenant_role       NOT NULL DEFAULT 'viewer',
  membership_status  public.membership_status NOT NULL DEFAULT 'active',
  invited_at         timestamptz,
  accepted_at        timestamptz,
  suspended_at       timestamptz,
  revoked_at         timestamptz,
  updated_at         timestamptz NOT NULL DEFAULT now(),

);

CREATE INDEX tenant_members_user_idx           ON public.tenant_members(user_id);
CREATE INDEX tenant_members_tenant_idx         ON public.tenant_members(tenant_id);
CREATE INDEX tenant_members_active_lookup_idx  ON public.tenant_members(user_id, tenant_id, membership_status);

CREATE TRIGGER tg_tenant_members_set_updated_at
  BEFORE UPDATE ON public.tenant_members
  FOR EACH ROW EXECUTE FUNCTION public.tg_set_updated_at();
```

Enums:

```sql
CREATE TYPE public.tenant_role
  AS ENUM ('owner','admin','manager','broker','captador','secretaria','viewer');

CREATE TYPE public.membership_status
  AS ENUM ('active','invited','suspended','revoked');
```

### 4.2 Grants

```sql
GRANT SELECT ON public.tenant_members TO authenticated;
GRANT ALL    ON public.tenant_members TO service_role;
-- (sem GRANT para anon)
```

### 4.3 RLS

```sql
ALTER TABLE public.tenant_members ENABLE ROW LEVEL SECURITY;

-- SELECT
CREATE POLICY tm_select ON public.tenant_members
  FOR SELECT TO authenticated
  USING (
    public.is_super_admin()
    OR user_id = auth.uid()
    OR public.user_belongs_to_tenant(tenant_id)
  );

-- FOR ALL (permissive, default)
CREATE POLICY tm_write ON public.tenant_members
  FOR ALL TO authenticated
  USING      (public.is_super_admin())
  WITH CHECK (public.is_super_admin());
```

Ambas as policies são **PERMISSIVE** (default do Postgres — `CREATE POLICY`
sem cláusula `AS RESTRICTIVE`).

### 4.4 Efeito real (policy + grants)

| Role            | SQL privilege efetivo em `tenant_members`             | Passa por RLS? |
|-----------------|--------------------------------------------------------|----------------|
| `anon`          | nenhum (sem GRANT)                                     | irrelevante — negado antes da RLS |
| `authenticated` | apenas `SELECT` (GRANT restrito a SELECT)              | sim — `tm_select` |
| `authenticated` | `INSERT/UPDATE/DELETE` — **não concedido**             | RLS `tm_write` seria avaliada, mas nunca alcança porque o GRANT bloqueia primeiro |
| `service_role`  | `ALL` (GRANT ALL); RLS **bypassada** por definição do service role | — |

Consequência arquitetural crítica: **hoje, um usuário `authenticated` não
consegue mutar `tenant_members` diretamente** por falta de GRANT, mesmo com
a policy `tm_write`. Qualquer futura RPC `SECURITY DEFINER` roda como owner
e ignora essa proteção — portanto o boundary de segurança da SCP-012 não
pode depender de `tm_write` isolada; precisa depender do desenho da RPC /
server fn (GRANT EXECUTE restrito + verificação explícita de autorização).

## 5. Autoridade comercial atual

A decisão comercial existe hoje inteiramente em TypeScript, server-side,
read-only, atrás de `requireTenant`:

| Módulo | Responsabilidade |
|---|---|
| `feature-catalog.ts` | catálogo canônico de feature keys + `evaluateFeatureCatalogGate` (short-circuit `not_evaluated`) |
| `feature-gate.ts` | `decideCommercialFeature` (entitled / not_entitled / billing_* / not_evaluated), reasons e source enum |
| `read-models.ts` | `deriveCommercialSummary`, `deriveEntitlementSnapshot`, `deriveBillingHealth`, projeções DTO whitelisted |
| `limit-decision.ts` | `normalizeCommercialSeatLimitInput` (strict input), tipos `CommercialLimitDecision` |
| `seat-limit-runtime.ts` | orquestração: catalog gate → feature decision → extração de limite → COUNT via reader |
| `seat-usage-reader.ts` | única leitura autoritativa `tenant_members COUNT` (`membership_status IN ('active','invited')`) |
| `commercial.functions.ts` | server functions (`getCommercialFeatureDecision`, `getCommercialSeatLimitDecision`, etc.) — loader único de contexto (`loadTenantCommercialContext`) |

Tabelas consultadas (todas via `supabaseAdmin`, exclusivamente SELECT):

- `tenant_subscriptions` — status/priority, `plan_id`;
- `commercial_plans` — nome/status do plano ativo;
- `commercial_plan_entitlements` — valores por plano;
- `tenant_entitlements` — overrides por tenant, com precedência;
- `tenant_billing_provider_mappings` — provider ativo;
- `billing_events` — último evento (apenas `received_at`, `processing_status`);
- `tenant_members` — COUNT com filtro `membership_status IN ('active','invited')`.

Precedência de entitlement (SCP-004+): `tenant_entitlements` > plan
entitlements > default (`source ∈ { 'tenant','plan','default','none' }`).

Extração de `users.seats`: dentro de `seat-limit-runtime.ts`, extraído do
`TenantEntitlementSnapshot`. Se o valor estiver ausente / inválido / não
efetivo, o short-circuit (SCP-011.2) impede leitura de `tenant_members`.

Nada disso participa hoje de qualquer transação Postgres. Toda a lógica é
avaliada em processo Node/Worker; não há isolamento contra corridas entre
resolver e mutation, porque não há mutation.

## 6. Alternativas arquiteturais

### 6.1 Estratégia A — SQL como nova autoridade canônica

Um resolver `SECURITY DEFINER` em Postgres passa a ser a autoridade única
(`resolve_commercial_seat_decision(_tenant uuid)` retornando
`(featureDecision, limitDecision)`). Runtime TypeScript delega ao resolver
SQL. Helpers TS permanecem apenas como tipos/validação (ou são
aposentados).

- Consistência: máxima (mesmo plano transacional que a mutation);
- Risco de dual resolver: eliminável se TS for delegador estrito;
- Complexidade: alta (portar catalog gate + precedência + billing map +
  reasons closed enum);
- Impacto no runtime atual: alto — reescrita da autoridade;
- Compatibilidade Supabase: total (Postgres nativo);
- Segurança: exige `search_path` fixo, `REVOKE ALL FROM PUBLIC/anon/authenticated`,
  `GRANT EXECUTE` restrito, identifiers qualificados, zero SQL dinâmico;
- Testabilidade: pgTAP + suíte de paridade cross-linguagem;
- Rollback: reversível (drop function); mas o runtime que passa a delegar
  fica indisponível sem ela;
- Operabilidade: mudanças de regra passam a exigir migration.

### 6.2 Estratégia B — Persistência de snapshot autoritativo

A decisão comercial é materializada previamente em uma tabela autoritativa
(ex.: `tenant_commercial_state` com `seat_limit`, `feature_gates`,
`source`, `reason`, `effective_at`, `version`) atualizada por processo
determinístico (webhook billing / recomputação agendada / trigger sobre
`tenant_entitlements` etc.). Enforcement lê a mesma linha `FOR UPDATE`.

- Consistência: alta se e somente se a materialização não fica stale;
- Risco de dual resolver: alto (TS ainda decide, materialização copia);
- Complexidade: alta (invalidação, versionamento, consistência eventual);
- Impacto: cria nova tabela autoritativa e novo caminho de escrita;
- Compatibilidade Supabase: total;
- Segurança: RLS + grants na nova tabela; escrita restrita ao processo de
  materialização;
- Testabilidade: paridade + testes de staleness;
- Rollback: mais complexo (tabela e produtor precisam sair juntos);
- Operabilidade: exige monitoração de defasagem.

### 6.3 Estratégia C — Conexão Postgres transacional server-side

Server functions abrem uma conexão direta (bypass PostgREST), fazem
`BEGIN`, executam o resolver comercial **e** a mutation de `tenant_members`
na mesma transação. O resolver comercial pode ainda ser SQL (A) ou uma
função server-side que roda dentro da transação e emite as queries via a
mesma conexão.

- Consistência: alta (mesma transação);
- Risco de dual resolver: médio se o resolver TS continuar autoritativo;
  eliminável se combinada com A;
- Complexidade: alta (gestão de pool, conexão dedicada, timeouts, retry);
- Impacto: introduz um caminho de acesso paralelo ao PostgREST;
- Compatibilidade Supabase: possível via connection string direta, mas foge
  do modelo padrão do projeto;
- Segurança: exige credenciais separadas + isolamento estrito;
- Testabilidade: exige Postgres real em CI para concorrência;
- Rollback: reversível;
- Operabilidade: adiciona superfície operacional (pool, health, limits).

## 7. Estratégia recomendada

**Estratégia A + Estratégia C parcial**: adotar SQL como autoridade única
(A) e executar a mutation dentro da **mesma transação** que invoca o
resolver, sem introduzir uma segunda conexão paralela — a mutation entra
no Postgres pela server function autorizada (SCP-012.0.3) e chama o
resolver SQL como parte da mesma transação (`BEGIN … resolve … lock …
mutate … COMMIT`).

Motivos:

- consistência transacional máxima sem dual authority;
- elimina o risco de resolver TS e resolver SQL divergirem — o TS delega;
- não introduz um novo caminho de acesso (mantém PostgREST + RPC como
  superfície);
- rollback claro (a RPC pode ser dropada e o runtime volta ao estado
  read-only atual, que é o comportamento hoje aprovado);
- Estratégia B fica **rejeitada** para o pré-requisito da SCP-012 porque
  materialização introduz staleness e cria dual authority por construção.

## 8. Plano de autoridade única

Para provar autoridade única (sem dual resolver):

1. o runtime TS de decisão comercial passa a **delegar** a
   `resolve_commercial_seat_decision(_tenant)`;
2. os helpers puros em `feature-catalog.ts`, `feature-gate.ts`,
   `read-models.ts`, `limit-decision.ts` permanecem apenas como:
   - contrato de tipos (`CommercialFeatureDecision`, `CommercialLimitDecision`,
     enums de reason/source);
   - normalização de input estrito;
   - projeção DTO whitelisted;
3. `seat-limit-runtime.ts` e `commercial.functions.ts` **deixam de**
   recomputar precedência/billing/catalog em TS após a materialização
   (SCP-012.0.2). Antes disso, coexistência é proibida — a transição
   ocorre em um único passo lógico, sem período de dual authority.

## 9. Membership Mutation Boundary

Boundary server-side futuro (não implementado nesta etapa). Operações
mínimas:

- `createMembership`
- `inviteMembership`
- `changeMembershipStatus` (transições dentro do domínio de
  `membership_status`)
- `changeMembershipRole`
- `suspendMembership`
- `revokeMembership`
- `reactivateMembership`
- `deleteMembership` (se e somente se o domínio autorizar)

Input permitido em cada mutation:

- `targetUserId`
- `targetStatus` (para transições)
- `targetRole` (para mudança de role)

Input **proibido** (nunca aceito do client):

- `tenantId`
- `limit`, `used`, `remaining`, `source`
- qualquer `CommercialLimitDecision` ou `CommercialFeatureDecision`
- `billingStatus`
- `seatDelta`
- `featureKey`

O `tenantId` vem exclusivamente de `requireTenant` (server-side).

## 10. Locking e transação

Comparação:

| Estratégia | Granularidade | Deadlocks | Cross-tenant | Retry | Observabilidade |
|---|---|---|---|---|---|
| `SELECT FOR UPDATE` em `public.tenants` (linha do tenant alvo) | por tenant | baixo se ordem consistente | isolado | simples | boa (linha real) |
| `pg_advisory_xact_lock(hashtext(tenant_id::text))` | por tenant (virtual) | baixo | isolado | simples | limitada (sem linha) |
| Tabela dedicada de lock/versionamento | por tenant | baixo | isolado | simples | ótima (versão explícita) |

**Recomendação**: `SELECT FOR UPDATE` sobre `public.tenants` pelo
`tenant_id` da requisição, no início da mutation, **antes** do resolver
SQL. Motivos:

- ordem canônica única (sempre `tenants` primeiro) elimina deadlocks
  cross-mutation;
- serializa mutations concorrentes do mesmo tenant sem afetar outros;
- não requer nova tabela;
- fácil de observar via `pg_locks`.

## 11. Segurança

Para a primitive SQL futura:

- `SECURITY DEFINER` apenas se necessário; caso contrário `INVOKER`;
- `SET search_path = public, pg_temp` explícito;
- identifiers sempre qualificados (`public.tenant_members`, etc.);
- `REVOKE ALL ON FUNCTION … FROM PUBLIC, anon, authenticated`;
- `GRANT EXECUTE ON FUNCTION … TO service_role` (e apenas ao papel que
  irá invocar a RPC — nunca `authenticated` direto);
- sem SQL dinâmico (`EXECUTE format(...)`);
- não retorna linhas internas — apenas o DTO whitelisted equivalente ao
  contrato atual;
- validação estrita de input (enum types + `IS NOT NULL` sobre
  `_tenant`);
- proibição de chamada direta do frontend garantida pelo
  `REVOKE`/`GRANT` acima.

## 12. Plano de paridade

Matriz de paridade obrigatória entre autoridade TS atual (SCP-011) e
autoridade SQL futura:

| Cenário | Feature decision | Limit decision | Reason | Source |
|---|---|---|---|---|
| feature fora do catálogo | `not_evaluated` | `not_evaluated` | `feature_not_cataloged` | `none` |
| entitled + limite ausente | `entitled` | `not_evaluated` | `limit_unavailable` | `none` |
| entitled + limite inválido | `entitled` | `not_evaluated` | `limit_invalid` | `none` |
| entitled + limite = 0 | `entitled` | `limit_reached` | `limit_reached` | correspondente |
| entitled + used < limit | `entitled` | `allowed` | `within_limit` | correspondente |
| entitled + used = limit | `entitled` | `limit_reached` | `limit_reached` | correspondente |
| entitled + used > limit | `entitled` | `limit_reached` (bounded) | `limit_reached` | correspondente |
| not_entitled | `not_entitled` | não avaliado (sem leitura de uso) | `not_entitled` | `none` |
| billing_unknown | `billing_unknown` | não avaliado | correspondente | `none` |
| billing_attention_required | `billing_attention_required` | avaliado normalmente | correspondente | correspondente |
| billing_blocked | `billing_blocked` | não avaliado | correspondente | `none` |
| source tenant / plan / default / none | preservados | preservados | preservados | preservados |

Regras de paridade:

- **mesmos inputs → mesmos outputs**, incluindo strings de reason;
- **mesma ordem de precedência** `tenant > plan > default > none`;
- **mesmos nulls / null-shapes** no DTO;
- feature negativo **nunca** consulta `tenant_members`;
- limite ausente / inválido **nunca** consulta `tenant_members`;
- catalog gate negativo **nunca** carrega contexto comercial.

A transição runtime→SQL só é aceita quando a suíte de paridade estiver
100% verde, sem dual authority intermediário.

## 13. Estratégia de testes

- **unitários TS**: continuam verdes contra o contrato atual;
- **pgTAP / SQL unit**: cobrem cada branch do resolver SQL;
- **paridade cross-linguagem**: fixtures compartilhadas alimentam TS
  (legado) e SQL (novo), comparando outputs byte-a-byte enquanto
  ambos coexistirem no branch de paridade;
- **integração**: server fn invocando resolver SQL, sob `requireTenant`;
- **boundary**: cada mutation de membership recusa `tenantId`, `limit`,
  `seatDelta`, `featureKey` no input;
- **autorização**: matriz roles × operações;
- **RLS/grants**: verificar que `authenticated` não pode `INSERT/UPDATE/
  DELETE` diretamente em `tenant_members`; que a RPC não é `EXECUTE`
  para `anon`/`authenticated`;
- **rollback**: drop da RPC → runtime cai para read-only aprovado;
- **idempotência**: repetir mutation não altera `used`;
- **concorrência real (Postgres real, dois clientes)**:
  - duas mutations +1 concorrentes com `limit=used`: exatamente uma
    passa, outra vê `limit_reached`;
  - N mutations concorrentes com `limit − used = k`: exatamente `k`
    passam;
- **cross-tenant**: mutation de tenant A não bloqueia tenant B;
- **regressão SCP-011**: todos os specs da cadeia SCP-011 continuam
  verdes sem mudança.

Ambiente: Postgres real (não mock) é obrigatório para concorrência.
