# SCP-012.0.3 — Membership Mutation Boundary Planning & Materialization

**Status:** Ready for External Audit
**Phase:** 04 — SaaS Commercial Platform
**Depends on:** SCP-012.0.2, SCP-012.0.2.1, SCP-012.0.2.2 (Accepted)
**Blocks:** SCP-012 (Blocked until acceptance)

## 1. Inventário inicial (preflight confirmado)

- Zero mutation runtime de `public.tenant_members` no código (apenas `.select()`).
- `authenticated` possuía privilégios de tabela `INSERT/UPDATE/DELETE` gated por RLS `tm_write` que exige `is_super_admin()`; esta etapa **revoga** os grants para alinhar com o invariante `SELECT-only`.
- `service_role` mantém acesso administrativo total.
- Nenhum fluxo runtime de invitation, aceite ou criação de `auth.users` associado a membership.
- Nenhuma remoção física runtime.
- `membership_status`: `active | invited | suspended | revoked` (preservado).
- `tenant_role`: `owner | admin | manager | broker | captador | secretaria | viewer` (preservado).
- Fontes tipadas: `src/integrations/supabase/membership-types.ts` e `membership-validation.ts` (mantidas).

## 2. Catálogo final de operações

`create_membership`, `change_role`, `suspend`, `reactivate`, `revoke`.

Contratos e regras de cada operação materializados na primitive SQL e no validator TypeScript conforme spec da SCP-012.0.3 (seções 3.1–3.5).

## 3. Operações explicitamente excluídas

Invite por e-mail, accept invitation, create auth user, delete físico, ownership transfer, criação de segundo owner, mudança para/de owner, mutation genérica, bulk, frontend, envio de e-mail, provider, billing enforcement.

`invited` permanece no domínio por compatibilidade — nenhum novo produtor é criado nesta etapa.

## 4. Contrato público

Union discriminada `MembershipMutationInput` em `src/lib/api/commercial/membership-mutation-types.ts`. `NonOwnerTenantRole` é derivado de `TENANT_ROLES` (exclui apenas `owner`).

### Campos proibidos (rejeitados pelo validator)

`tenantId`, `actorUserId`, `tenantOrigin`, `isSuperAdmin`, `isOwner`, `targetStatus`, `membershipStatus`, `seatDelta`, `featureKey`, `allowed`, `reason`, `source`, `limit`, `used`, `remaining`, `billingStatus`, `CommercialLimitDecision`, `CommercialFeatureDecision`.

O validator também rejeita **qualquer** propriedade adicional não listada no shape esperado.

## 5. Trusted Actor Context

`actorUserId`, `tenantId` e `tenantOrigin` são derivados server-side por `requireSupabaseAuth` + `requireTenant`. Nunca aceitos do payload. O header `x-tenant-id` continua sendo apenas transporte; a autoridade real vem da revalidação em `get_current_tenant_id()` e é reconfirmada dentro da primitive SQL.

## 6. Matriz de autorização

- **Usuário comum:** só executa quando possui `tenant_role = 'owner'` + `membership_status = 'active'` + `is_owner = true` no tenant. Origens válidas: `selection` ou `single-membership`. Nenhuma outra role tenant (incluindo `admin`) recebe autoridade nesta etapa.
- **Super Admin:** exige `super_admin` em `public.user_roles`, `tenantOrigin = 'impersonation'` e tenant alvo existente. Rejeita `selection`/`single-membership`.
- **Proteção de owner:** qualquer target com `tenant_role = 'owner'` OU `is_owner = true` é rejeitado. Elimina necessidade imediata de last-owner check e ownership transfer.

## 7. Transições materializadas

| Operação            | Prévio → Novo                                        |
|---------------------|------------------------------------------------------|
| `create_membership` | (nenhum) → `active`                                  |
| `change_role`       | qualquer não-revoked → mesmo status, nova role       |
| `suspend`           | `active` → `suspended`                               |
| `reactivate`        | `suspended` → `active` (preserva joined_at)          |
| `revoke`            | `active` / `invited` / `suspended` → `revoked`       |

Repetir o estado atual retorna `changed = false` sem lançar erro. `revoked` é terminal nesta etapa. `change_role` em membership `revoked` é rejeitado.

## 8. Seat delta interno

Helper puro `classifyMembershipSeatDelta` em `membership-seat-delta.ts`.

| Operação          | Prévio → Novo             | Delta |
|-------------------|---------------------------|-------|
| create_membership | (nenhum) → active         | +1    |
| reactivate        | suspended → active        | +1    |
| suspend           | active → suspended        | -1    |
| revoke            | active → revoked          | -1    |
| revoke            | invited → revoked         | -1    |
| revoke            | suspended → revoked       |  0    |
| change_role       | (qualquer)                |  0    |
| qualquer noop     | (não `changed`)           |  0    |

O `seatDelta` **não** é aceito do client, **não** é exposto no DTO público de retorno desta etapa, **não** autoriza a mutation, **não** chama o resolver comercial e **não** bloqueia por limite.

## 9. Primitive SQL

Assinatura real (aplicada em migration `supabase/migrations/20260713212535_fdf03e63-87cb-4288-9806-09ff0368747c.sql`; ACL da tabela corrigida em `supabase/migrations/20260713213947_4b389fff-0a89-446d-9cf7-753b21a79586.sql`):

```sql
public.mutate_tenant_membership(
  _actor_user_id  uuid,
  _tenant_id      uuid,
  _tenant_origin  text,
  _operation      text,
  _target_user_id uuid,
  _target_role    text DEFAULT NULL
) RETURNS jsonb
LANGUAGE plpgsql VOLATILE SECURITY DEFINER
SET search_path = public, pg_temp
```

Zero SQL dinâmico, zero `EXECUTE format`, identifiers qualificados. Não aceita `tenantLimit`, `seatDelta`, `commercialDecision`, `billingStatus`, `featureKey`.

### ACL final da função (verificada em `pg_proc.proacl`)

```
mutate_tenant_membership | postgres     | EXECUTE
mutate_tenant_membership | service_role | EXECUTE
```

Sem grant para `PUBLIC`, `anon` ou `authenticated`.

### ACL final da tabela `public.tenant_members`

Verificada via `pg_class` + `aclexplode` + `pg_get_userbyid` na migration corretiva:

```
authenticated | SELECT
service_role  | SELECT
service_role  | INSERT
service_role  | UPDATE
service_role  | DELETE
service_role  | TRUNCATE
service_role  | REFERENCES
service_role  | TRIGGER
service_role  | MAINTAIN
```

Invariante final:

- **PUBLIC** — nenhum privilégio explícito.
- **anon** — nenhum privilégio explícito.
- **authenticated** — `SELECT` somente.
- **service_role** — privilégios administrativos (`SELECT/INSERT/UPDATE/DELETE` + defaults do Data API).

A migration corretiva contém assertion `DO $$ ... $$` baseada em `pg_class`/`aclexplode`/`pg_get_userbyid` que aborta se qualquer parte do invariante for violada (sem `EXCEPTION WHEN OTHERS`, sem `RAISE NOTICE`).

## 10. Boundary TypeScript

`src/lib/api/commercial/membership-mutation-boundary.server.ts` (`*.server.ts`, bloqueado do bundle do cliente):

- Valida `MembershipMutationInput` via `parseMembershipMutationInput`.
- Recebe `TrustedActorContext` já produzido pelos middlewares.
- Chama **exclusivamente** `supabaseAdmin.rpc("mutate_tenant_membership", ...)`.
- Valida o retorno com `validateMembershipMutationResult(raw, expected)` — validator **semântico** que exige correspondência de `tenantId`, `targetUserId`, `operation` com o contexto e enforce das transições permitidas por operação (incluindo estabilidade de role em `suspend/reactivate/revoke` e proteção absoluta de owner no retorno).
- Falha deterministicamente em erro RPC (`membership_mutation_rpc_failed`) ou desvio semântico.
- Sem fallback TypeScript. Sem escrita direta em `tenant_members`.

## 11. Testes

### Unit (`run-tenant-specs.ts`, exit 0)

35 casos em `membership-mutation-input.spec.ts` cobrindo cada operação, campos proibidos individualmente, roles inválidas, `owner` rejeitada, união discriminada, DTO validator (shape + enums), semântica (`tenantId/targetUserId/operation` mismatch, `role=owner`/`previousRole=owner`, `create_membership` com `previousStatus` não nulo ou `status` diferente de `active`, `change_role` alterando status, `changed=false` alterando role, `changed=true` sem alterar role, transições inválidas de `suspend/reactivate/revoke`, `role` alterada em operações de status) e seat delta em cada transição.

### Integração crítica (`run-membership-mutation-parity-specs.ts`, exit 0)

14 cenários contra Postgres real com fixtures isoladas (auth users criados via Admin API + tenant sintético), cobrindo: criação, change_role idempotente, suspend, reactivate, revoke, rejeição de não-owner, super_admin com/sem impersonação, target owner rejeitado e três **provas separadas de privilege boundary**:

- **anon** — client sem sessão: RPC rejeitada + `INSERT` direto em `tenant_members` rejeitado.
- **authenticated** — usuário temporário criado como fixture, `signInWithPassword`, `getUser()` confirma identidade, RPC rejeitada + `INSERT` direto rejeitado.
- **service_role** — mutations autorizadas continuam funcionando.

Cleanup **fail-closed** executado no `finally`: deleta memberships, `user_roles` temporários, tenant e todos os auth users criados; verifica ausência de resíduo (tenant, memberships, user_role, auth users). Qualquer erro/resíduo → `exit != 0`.

## 12. Ausências obrigatórias e confirmadas

- **Zero UI/frontend.** Boundary é server-only; nenhuma rota, tela, botão ou hook adicionado.
- **Zero lock.** Sem `SELECT ... FOR UPDATE`, sem `pg_advisory_xact_lock`.
- **Zero commercial enforcement.** Sem chamada a `resolve_commercial_seat_decision`, sem verificação de `users.seats`, sem `CommercialLimitDecision` no retorno.
- **Zero invitation flow / accept flow.**
- **Zero criação de auth user no runtime de produto.** O harness de integração cria usuários temporários exclusivamente como fixtures e os remove de forma fail-closed.
- **Zero DELETE físico de membership.**
- **Zero owner mutation.**
- **Zero grant de escrita a `authenticated` / `anon`** em `tenant_members` (invariante ACL agora fechado e validado por assertion).

## 13. Riscos e limitações reais

- Boundary materializado e testável, porém **sem produtor no runtime de produto**. Nenhum fluxo operacional o consome nesta etapa — intencional: enforcement atômico com o resolver comercial e lock de tenant é responsabilidade da SCP-012 principal.
- `invited` permanece no domínio sem produtor novo; será revisitado se/quando invitation flow for materializado (fora deste escopo).

## 14. Próximo gate
