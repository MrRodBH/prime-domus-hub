# SCP-012.0.1.3 — Server-Only RPC Trusted Actor Context & Hard Gate S0 Contract Reconciliation

## Status

Ready for External Audit

## 1. Problema confirmado

A tentativa de execução da SCP-012.0.2 foi corretamente interrompida no
Hard Gate S0 com o status:

```
SCP-012.0.2 = Blocked Before Implementation —
Authenticated Identity / Server-Only EXECUTE Contract Incompatibility
```

O contrato herdado da SCP-012.0 exigia simultaneamente três condições
incompatíveis com a infraestrutura atual:

1. `auth.uid()` obrigatório como identidade do ator dentro da futura
   RPC comercial;
2. `REVOKE EXECUTE` de `authenticated` sobre a RPC;
3. `GRANT EXECUTE` somente para `service_role`.

Nenhum dos dois clients Supabase existentes satisfaz simultaneamente
`auth.uid()` genuíno **e** identidade Postgres `service_role`:

- `supabaseAdmin` (`src/integrations/supabase/client.server.ts`):
  usa `SUPABASE_SERVICE_ROLE_KEY`, é server-only, não transporta o JWT
  do usuário final; role Postgres efetiva = `service_role`;
  `auth.uid()` não representa o usuário autenticado.
- `context.supabase` (produzido por `requireSupabaseAuth`,
  `src/integrations/supabase/auth-middleware.ts`): usa
  `SUPABASE_PUBLISHABLE_KEY` + bearer token do usuário; role Postgres
  efetiva = `authenticated`; `auth.uid()` representa o usuário; exigiria
  `GRANT EXECUTE` para `authenticated`, contradizendo o boundary
  service_role-only.

O repositório não expõe uma terceira identidade Postgres capaz de
transportar o JWT do usuário e ao mesmo tempo executar como
`service_role`. A criação de tal mecanismo (custom JWT + custom role,
capability token assinado, conexão Postgres direta) não é pré-requisito
da SCP-012 no threat model atual.

## 3. Matriz de alternativas

| # | Nome                                                | auth.uid() genuíno | EXECUTE service_role-only | Complexidade    | Superfície criptográfica | Adotada? |
|---|-----------------------------------------------------|--------------------|---------------------------|-----------------|--------------------------|----------|
| A | service_role-only + Trusted Actor Context           | não (server-attested) | sim                    | baixa           | nenhuma nova             | **sim**  |
| B | RPC para `authenticated` usando `auth.uid()`        | sim                | não (grant a authenticated) | baixa         | nenhuma nova             | não      |
| C | Custom JWT + custom Postgres role                   | sim (custom)       | sim (custom role)         | alta            | signing keys, rotação    | não      |
| D | Capability token assinado (HMAC/assimétrico)        | não                | sim                       | média-alta      | secret compartilhado, Vault | não   |
| E | Conexão Postgres direta com `SET LOCAL ROLE`        | não                | n/a (bypass PostgREST)    | alta            | pool, transaction callback | não    |

### Alternativa A — service_role-only com Trusted Actor Context (recomendada)

Fluxo:

```
browser
  → server function autenticada
    → requireSupabaseAuth (valida JWT, produz context.userId)
    → requireTenant (produz tenantId, tenantOrigin)
    → deriva TrustedCommercialActorContext internamente
    → supabaseAdmin.rpc(fn, { actor, tenant, origin, requestedIncrement })
      → RPC service_role-only
        → revalidação SQL do contexto contra estado persistente
        → decisão comercial + (futura) mutation dentro da transação
```

Contexto conceitual (não implementado nesta etapa):

```
type TrustedCommercialActorContext = {
  actorUserId: string;
  tenantId: string;
  tenantOrigin: "impersonation" | "selection" | "single-membership";
};
```

Características:

- `actorUserId`, `tenantId`, `tenantOrigin` **não** vêm do payload
  público;
- todos são derivados server-side pelos middlewares existentes;
- a RPC não possui `EXECUTE` para `authenticated`;
- a RPC é chamada exclusivamente por `service_role`;
- Postgres revalida actor, tenant, membership e origem contra o estado
  persistente.

### Alternativa B — RPC para `authenticated` com `auth.uid()`

- preserva `auth.uid()` como identidade;
- exige `GRANT EXECUTE` a `authenticated`;
- permite chamada direta do browser;
- contradiz o boundary "sem acesso comercial direto pelo client" fixado
  pela SCP-012.0 §6/§8.

Recomendação: **rejeitada**.

### Alternativa C — Custom JWT + custom Postgres role

- emissão server-side de JWT com claim `sub` customizado;
- role Postgres dedicada consumindo esse JWT via PostgREST;
- gestão de signing keys, rotação, revogação;
- superfície criptográfica nova sem ganho de segurança sobre A no
  threat model atual (o servidor autenticado já é boundary confiável).

Recomendação: **não selecionada** para a SCP-012.

### Alternativa D — Capability token assinado

- token de curta duração emitido pela server function e consumido pela
  RPC;
- HMAC ou assinatura assimétrica com secret em Vault;
- adiciona defesa em profundidade contra chamadas indevidas de
  `supabaseAdmin` (nenhuma existe hoje fora do boundary de auth).

Recomendação: **reservada como defesa em profundidade futura**; não
necessária para o primeiro cutover.

### Alternativa E — Conexão Postgres direta

- pool próprio + transaction callback + `SET LOCAL ROLE`;
- contradiz a Estratégia A SQL/PostgREST já aceita na SCP-012.0 §8;
- introduz novo canal de conexão fora do PostgREST.

Recomendação: **rejeitada**.

## 4. Decisão recomendada

**Alternativa A — service_role-only com Trusted Actor Context.**

A decisão preserva:

- a Estratégia A SQL/PostgREST canônica da SCP-012.0;
- o boundary "sem acesso comercial direto pelo client";
- o contrato canônico `CommercialLimitDecision` (SCP-012.0.1);
- o short-circuit `billing_attention_required` (SCP-012.0.1);
- a authoritativeness Postgres da contagem e da decisão;
- a integridade transacional futura da mutation de `tenant_members`.

## 5. Contrato público (server function)

A server function pública aceitará **somente**:

```
undefined
{}
{ requestedIncrement: 1 }
```

Proibidos no payload público (rejeitados pelo `inputValidator`):

```
actorUserId
tenantId
tenantOrigin
isSuperAdmin
impersonation
featureKey
limit
used
remaining
source
billingStatus
CommercialLimitDecision
CommercialFeatureDecision
```

## 6. Contexto server-side interno

A server function derivará internamente e transportará como parâmetros
**internos** da RPC (não expostos ao client):

- `actorUserId = context.userId` (produzido por `requireSupabaseAuth`);
- `tenantId = context.tenant.tenantId` (produzido por `requireTenant`);
- `tenantOrigin = context.tenant.origin` (produzido por
  `requireTenant`).

Justificativa da segurança do transporte:

- não são aceitos pelo `inputValidator` da server function;
- não são controlados pelo browser;
- são derivados de JWT previamente validado por `requireSupabaseAuth`;
- são derivados do estado de seleção/impersonation por `requireTenant`;
- a RPC será executável **somente** por `service_role`.

Não transportados como parâmetros:

```
isSuperAdmin (boolean)
allowed
reason
limit
used
remaining
billing decision
```

O estado de Super Admin é derivado exclusivamente do estado persistente
dentro do SQL.

## 7. Revalidação SQL obrigatória (futuro)

A futura RPC deverá revalidar dentro da transação:

Invariantes gerais:

- `tenantId` existe em `public.tenants`;
- `actorUserId` não é nulo;
- actor existe em `auth.users`;
- `requestedIncrement = 1`.

Para usuário comum:

- actor **não** possui role `super_admin` em `public.user_roles`;
- `tenantOrigin ∈ { 'selection', 'single-membership' }`;
- existe `public.tenant_members` com:
  - `tenant_id = tenantId`;
  - `user_id = actorUserId`;
  - `membership_status = 'active'`.

Para Super Admin:

- actor possui role `super_admin` em `public.user_roles`;
- `tenantOrigin = 'impersonation'` (exatamente);
- tenant existe;
- **não** utilizar membership comum como substituto de impersonação.

Rejeitar (erro determinístico):

- Super Admin com `selection`;
- Super Admin com `single-membership`;
- usuário comum com `impersonation`;
- membership `suspended`;
- membership `revoked`;
- tenant inexistente;
- actor inexistente;
- origin inválida.

## 8. Fonte de confiança

- `auth.uid()` **não** é a fonte de identidade dentro da RPC
  service_role-only;
- fonte primária de autenticação: `requireSupabaseAuth`;
- fonte do tenant context: `requireTenant`;
- a RPC recebe contexto server-attested como parâmetros internos;
- Postgres revalida o contexto contra o estado persistente;
- o banco **não** confia no payload público.

## 9. Threat model

- o browser é não confiável;
- headers enviados pelo browser são apenas transporte;
- a server function autenticada é boundary confiável;
- `service_role` é segredo de backend;
- comprometimento da `service_role` representa comprometimento total do
  backend e **não** é mitigado por `auth.uid()`;
- a revalidação SQL protege contra erros de integração, contexto
  inconsistente e uso cross-tenant — não contra comprometimento
  integral do servidor.

## 10. Reconciliação do requisito `auth.uid()`

A regra herdada:

> `auth.uid()` obrigatório dentro da futura RPC.

é substituída, **apenas para a primitive RPC service_role-only da
cadeia SCP-012**, pela regra:

> Para RPC executável exclusivamente por `service_role`, `auth.uid()`
> não é utilizado como identidade do usuário final. A identidade do
> ator é derivada do JWT validado pela server function, transportada
> como contexto interno não controlável pelo client e revalidada no
> banco.

Escopo da substituição:

- aplica-se **apenas** à primitive RPC comercial futura da SCP-012;
- **não** altera RLS autenticada existente;
- **não** altera policies que usam `auth.uid()`;
- **não** altera funções invocadas diretamente por `authenticated`;
- **não** altera `requireSupabaseAuth` nem `auth-attacher`.

## 11. Contrato de grants (documentado, não implementado)

```sql
REVOKE ALL ON FUNCTION <assinatura_real> FROM PUBLIC;
REVOKE ALL ON FUNCTION <assinatura_real> FROM anon;
REVOKE ALL ON FUNCTION <assinatura_real> FROM authenticated;

GRANT EXECUTE ON FUNCTION <assinatura_real> TO service_role;
```

- o controle que impede chamada direta pelo frontend é o privilege
  boundary da função;
- `actorUserId` **não** substitui o privilege boundary;
- a função **não** poderá ser executada por `authenticated`;
- a função **não** será protegida apenas por obscuridade.

## 12. Impacto sobre SCP-012.0.2

- SCP-012.0.2 permanece bloqueada até aceitação externa da
  SCP-012.0.1.3;
- a materialização SQL deverá seguir integralmente §5–§11 deste
  documento;
- o `inputValidator` da server function pública permanecerá restrito ao
  contrato §5;
- o SQL revalidará §7 antes de qualquer decisão ou mutation;
- a decisão comercial e a mutation futura de `tenant_members` deverão
  ocorrer dentro da mesma transação Postgres, invocadas por
  `supabaseAdmin.rpc(...)`.

## 13. Riscos

- erro de integração que envie contexto inconsistente à RPC → mitigado
  pela revalidação SQL (§7);
- uso indevido de `supabaseAdmin` fora do boundary de auth → mitigado
  pelo policiamento estático (`client.server.ts` restrito a
  `.server.ts` / handlers) e por revisão de código;
- comprometimento da `service_role` → fora do escopo do controle
  aplicativo (item de threat model, §9);
- desvio conceitual de que `actorUserId` seria "identidade Postgres" →
  mitigado por documentação explícita (§8, §10).
