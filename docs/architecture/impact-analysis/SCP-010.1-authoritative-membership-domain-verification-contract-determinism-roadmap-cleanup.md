# SCP-010.1 — Authoritative Membership Domain Verification, Contract Determinism & Roadmap Cleanup

## Status

Ready for External Audit

## 1. Objetivo

Registrar a verificação autoritativa do domínio de membership (schema,
enums, constraints, colunas), da unidade de contagem e da representação
de convites, deixando as decisões preparatórias determinísticas antes
de qualquer runtime da SCP-011. Esta etapa é exclusivamente documental.

## 2. Fontes inspecionadas

  `commercial_entitlement_definitions`.
- Schema ao vivo via `psql \d` das tabelas acima.

- Documentação F3.6 (domínio de membership publicado).

## 3. Evidências autoritativas

- `public.membership_status` = `active | invited | suspended | revoked`.
- `public.tenant_role` = `owner | admin | manager | broker | captador | secretaria | viewer`.
- `public.tenant_members.user_id` `NOT NULL`.
- `tenant_members_pkey PRIMARY KEY (tenant_id, user_id)` — total,
  sem predicado parcial.
- `public.tenant_members.membership_status` `NOT NULL DEFAULT 'active'`.
- `public.tenant_members.tenant_role` `NOT NULL DEFAULT 'viewer'`.
- `types.ts`, `membership-types.ts` e `membership-validation.ts`
  alinhados 1:1 ao banco. Nenhum domínio legado coexiste.

## 4. Unidade de contagem

```
COUNT(*)
FROM public.tenant_members
WHERE tenant_id = :tenantId
  AND membership_status IN ('active', 'invited')
```

- `active` consome assento.
- `invited` reserva assento.
- `suspended` não consome.
- `revoked` não consome.
- Todos os valores de `tenant_role` consomem quando o status é counted.
- `COUNT(DISTINCT user_id)` não é necessário: a PK composta
  `(tenant_id, user_id)` garante unicidade por linha.

## 5. Convites

- Não existe tabela separada de invitations.
- Não existe convite somente por e-mail.
- `invited` exige `user_id` vinculado a `auth.users` (coluna `NOT NULL`).
- Duplicidade é impedida pela PK `(tenant_id, user_id)`; reinvitar o
  mesmo `(tenant, user)` requer UPDATE na linha existente.

## 6. Limite users.seats

- `value_int` é `integer` não negativo (`CHECK value_int >= 0`) em
  `tenant_entitlements` e `commercial_plan_entitlements`.
- `limit = 0` é limite finito válido.
- `NULL` não significa unlimited.
- Ausência de linha não significa unlimited.
- `unlimited` não é suportado.
- `requestedIncrement` aceita somente `1`.

Redefinições de DTO, `reason`, `source` e integração comercial
não pertencem a esta SCP-010.1 e são tratadas pela SCP-010.2.

## 7. Relação com a SCP-010.2

Esta SCP-010.1 não substituiu integralmente a SCP-010.
