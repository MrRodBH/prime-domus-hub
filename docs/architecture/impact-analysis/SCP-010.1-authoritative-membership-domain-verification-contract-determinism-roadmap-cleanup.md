## 5. Conclusões autoritativas

### 5.1 Unicidade

`tenant_members_pkey PRIMARY KEY (tenant_id, user_id)` — total, sem
predicado parcial. Cobre reinvites e revogados: reinvitar o mesmo
`(tenant, user)` é impossível como nova linha; requer UPDATE na
linha existente. Nenhum índice adicional precisa ser criado para
garantir a unidade contada.

### 5.2 Convites

`user_id NOT NULL` e ausência de tabela separada de invitations →
convite só existe com identidade `auth.users` pré-criada. Não
existe convite somente-por-email nesta modelagem. A unidade contada
para `used` é `COUNT(*)` das linhas com
`membership_status ∈ {active, invited}` — sem necessidade de
`DISTINCT`.

### 5.3 Fonte do limite

Cadeia determinística `tenant_entitlements > commercial_plan_entitlements
via tenant_subscriptions vigente > not_evaluated`. Coluna numérica
= `value_int` (`integer NOT NULL` no check `>= 0`). UNIQUEs do banco
garantem cardinalidade única em ambos os níveis; violação de UNIQUE
observada em runtime é tratada como `not_evaluated`.

### 5.4 Unlimited

Nenhuma representação suportada. `NULL`, ausência, valor inválido e
cardinalidade quebrada → `allowed=false`, `reason=not_evaluated`.
`0` é limite finito válido.

### 5.5 Reason único para tenant sem assinatura

`not_evaluated`. `not_entitled`/`billing_unknown` eliminados.

### 5.6 `requestedIncrement`

Contrato único: default `1`, aceitável apenas `1`; qualquer outro
valor é rejeitado no boundary antes do DTO.
