# SCP-010.1 — Authoritative Membership Domain Verification, Contract Determinism & Roadmap Cleanup

## Status

Ready for External Audit.

## 1. Objetivo

Registrar a verificação autoritativa do domínio de membership (schema,
enums, constraints, colunas) e o fechamento determinístico das decisões
preparatórias da SCP-010 antes de qualquer runtime da SCP-011.

Esta etapa é exclusivamente documental. A consolidação final do DTO,
a correção de `reason`/`source`, a eliminação do dual-path de resolução
comercial e a substituição integral do documento principal da SCP-010
pertencem à SCP-010.2 (ver §7).

## 2. Fontes inspecionadas (autoritativas)

- Migrations em `supabase/migrations/` que definem enums, colunas,
  constraints e RLS de `public.tenant_members`, `tenant_subscriptions`,
  `tenant_entitlements`, `commercial_plan_entitlements` e
  `commercial_entitlement_definitions`.
- Schema ao vivo via `psql \d` das tabelas acima.
- Código: `src/integrations/supabase/membership-types.ts`,
  `src/integrations/supabase/membership-validation.ts`,
  `src/integrations/supabase/tenant-repository.ts`,

  `src/lib/api/tenant-selection.functions.ts`,
  `src/lib/api/super.functions.ts`,
  `src/lib/api/commercial/feature-catalog.ts`.
- Documentação F3.6 (domínio de membership publicado).

## 3. Evidências autoritativas

- `public.membership_status` (enum) = `active | invited | suspended | revoked`.
  `removed` não existe.
- `public.tenant_role` (enum) = `owner | admin | manager | broker | captador | secretaria | viewer`.
  `agent` e `guest` não existem. Coluna `membership_role` não existe.
- `public.tenant_members.membership_status` `NOT NULL DEFAULT 'active'`.
- `public.tenant_members.tenant_role` `NOT NULL DEFAULT 'viewer'`.
- `public.tenant_members.user_id` `uuid NOT NULL`.
- `tenant_members_pkey PRIMARY KEY (tenant_id, user_id)` — total, sem
  predicado parcial.
- `types.ts`, `membership-types.ts` e `membership-validation.ts`
  alinhados 1:1 ao banco. Nenhum domínio legado coexiste.

## 4. Unidade de contagem

A unidade contada para `used` é:

```
COUNT(*) FROM public.tenant_members
WHERE tenant_id = :tenant
  AND membership_status IN ('active', 'invited')
```

`COUNT(DISTINCT user_id)` não é usado. A PK `(tenant_id, user_id)`
garante que cada `(tenant, user)` aparece no máximo uma vez.

## 5. Convites

- Não existe tabela de invitations separada.
- Não existe convite somente por e-mail.
- `invited` exige `user_id` já vinculado a `auth.users` (coluna `NOT NULL`).
- Duplicidade é impedida pela PK `(tenant_id, user_id)`; reinvitar o
  mesmo `(tenant, user)` requer UPDATE na linha existente.

## 6. Limite (`users.seats`)

- Coluna numérica de origem: `value_int integer` com `CHECK value_int >= 0`
  em `tenant_entitlements` e `commercial_plan_entitlements`.
- `unlimited` não é suportado.
- `NULL` e ausência não significam ilimitado; resolvem para
  `allowed=false, reason=not_evaluated`.
- `limit = 0` é limite finito válido.
- `requestedIncrement` aceita somente `1`; qualquer outro valor é
  rejeitado no boundary antes do DTO.

Precedência de fontes (determinística):

1. `tenant_entitlements` vigente.
2. `commercial_plan_entitlements` via `tenant_subscriptions` vigente
   (`trialing, active, past_due, internal, demo`).
3. `not_evaluated`.

Cardinalidade única é garantida pelas UNIQUEs do banco em ambos os
níveis; violação observada em runtime → `not_evaluated`, sem
desempate por `LIMIT 1`/`ORDER BY`.

## 7. Relação com a SCP-010.2

- SCP-010.1 (esta etapa): verificação do domínio, schema, enums,
  constraints e unidade de contagem.
- SCP-010.2: alinhamento final do DTO `CommercialLimitDecision`,
  correção dos valores de `reason` e `source`, eliminação do
  dual-path de resolução comercial e substituição integral do
  documento principal da SCP-010.

A auditoria externa deve considerar SCP-010, SCP-010.1 e SCP-010.2
como cadeia única. Esta etapa não consolidou o DTO nem substituiu
a SCP-010.

## 8. Mutation inventory

Busca em todo o repositório por escritas em `tenant_members`
(`insert`, `update`, `upsert`, `delete`, `from("tenant_members")`,
RPCs, edge functions, seeds, admin screens):

| Operação | Arquivo | Boundary |
| --- | --- | --- |
| SELECT | `src/lib/api/tenant-selection.functions.ts` | server |
| SELECT (count) | `src/lib/api/super.functions.ts` | server admin |
| SELECT | `src/integrations/supabase/tenant-repository.ts` | server |

Nenhuma mutation aplicacional de `tenant_members` foi encontrada. RLS
restringe writes a `is_super_admin()`; alterações de membership hoje
ocorrem exclusivamente via SQL/admin. Consequência:

- SCP-011 entregará runtime **read-only** de avaliação, sem consumidor
  operacional que aumente `used`.
- Enforcement e mutations pertencem a etapa posterior (SCP-012 e
  seguintes), fora do escopo desta cadeia.
