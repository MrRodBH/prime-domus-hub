# SCP-012.0.1.3.1 — Trusted Context Provenance, Client Inventory & Git Evidence Lock

## Status

Accepted

## 1. Problema auditado

A auditoria externa considerou a arquitetura da SCP-012.0.1.3
tecnicamente coerente (service_role-only RPC + Trusted Actor Context
derivado server-side + revalidação SQL), mas identificou três
inconsistências exclusivamente documentais/evidenciais:

1. a evidência Git da materialização não continha SHA concreto do
   HEAD, apenas o placeholder "registrado no relatório de execução do
   agente após a materialização";
2. o inventário de clients Supabase incluía a linha
   `serverPublishable (tenant.server)` sem citação explícita do
   arquivo, da declaração `createClient` e dos consumidores reais;
3. a proveniência de `actorUserId`, `tenantId` e `tenantOrigin` era
   descrita de forma indiferenciada como "não controlada pelo
   browser", ocultando que `tenantId` pode chegar ao servidor como
   transporte não confiável (por exemplo via `x-tenant-id`) e que
   apenas `tenantOrigin` é integralmente calculado server-side.

Esta etapa corrige apenas essas três inconsistências. **Nenhuma
decisão arquitetural da SCP-012.0.1.3 é reaberta.** A Alternativa A
permanece selecionada.

## 5. Proveniência de `actorUserId`

- não aceito no `inputValidator` público da server function;
- derivado exclusivamente do JWT validado por `requireSupabaseAuth`
  (`src/integrations/supabase/auth-middleware.ts`);
- disponibilizado como `context.userId` para o corpo do handler;
- revalidado no PostgreSQL contra `auth.users` e (para papel comum)
  contra `tenant_members`.

## 6. Proveniência de `tenantId`

- **pode** ser indicado pelo client como transporte (por exemplo via
  header `x-tenant-id`);
- o valor bruto do transporte **não** é autoridade;
- `requireTenant` (`src/integrations/supabase/tenant-middleware.ts`)
  valida esse transporte contra o estado persistente — tenant
  existente, membership ativa do usuário, ou impersonação explícita
  autorizada;
- somente após a validação server-side o `tenantId` é considerado
  server-attested;
- o PostgreSQL revalida `tenantId` contra `public.tenants`,
  `public.tenant_members` e `public.user_roles` dentro da RPC futura.

## 7. Proveniência de `tenantOrigin`

- **nunca** aceito diretamente do client, nem como payload público nem
  como header;
- calculado server-side por `requireTenant` a partir do caminho
  validado de resolução do tenant, e restrito a
  `{ 'impersonation', 'selection', 'single-membership' }`;
- revalidado dentro da RPC futura: Super Admin exige
  `tenantOrigin = 'impersonation'`; usuário comum exige
  `tenantOrigin ∈ { 'selection', 'single-membership' }` com membership
  `active`.

## 8. Transporte × autoridade

| Campo | Transporte aceitável | Autoridade |
|-------|----------------------|------------|
| `actorUserId` | não aceito como transporte; derivado do JWT | `requireSupabaseAuth` + revalidação SQL |
| `tenantId` | `x-tenant-id` (client), somente como transporte não confiável | `requireTenant` + revalidação SQL |
| `tenantOrigin` | não aceito de nenhum transporte do client | `requireTenant` + revalidação SQL |

Regras normativas:

- `x-tenant-id` = transporte;
- `x-tenant-id` ≠ autoridade;
- server function autenticada = boundary autenticado;
- PostgreSQL = autoridade final da decisão comercial.

### Modelo de evidência não autorreferencial

Um arquivo versionado não deve registrar o SHA do próprio commit que
o contém: a inserção do SHA produz um novo commit e o SHA anterior
deixa de ser o HEAD final. Por essa razão, o HEAD do patch posterior
que finaliza esta etapa como `Accepted` (SCP-012.0.1.3.1.1) não é
persistido dentro dos documentos desta etapa; ele é apresentado no
relatório textual de execução e verificado externamente diretamente
no GitHub sobre o branch `main`.
