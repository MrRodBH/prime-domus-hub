### 3.2 Migration fail-closed de ACL (§15)

Aplicada em PostgreSQL real:

```
supabase/migrations/20260713203357_b2698312-fc50-4d3b-b9ba-9fa0cdba171a.sql
```

Conteúdo:

1. `REVOKE ALL` de `PUBLIC`, `anon`, `authenticated`;
2. `REVOKE ALL` de `sandbox_exec` quando a role existe — **sem** bloco
   `WHEN OTHERS`. Qualquer falha no REVOKE aborta a migration;
3. `GRANT EXECUTE` para `service_role`;
4. bloco `DO $$ … $$` que **falha com `RAISE EXCEPTION`** se, ao final,
   qualquer grantee diferente do owner (`postgres`) ou de
   `service_role` ainda detiver `EXECUTE`.

ACL antes:

```
{postgres=X/postgres, service_role=X/postgres, sandbox_exec=X/postgres}
```

ACL depois (verificado nesta execução):

```
{postgres=X/postgres, service_role=X/postgres}
```

- `has_function_privilege('anon',          …, 'EXECUTE') = false`
- `has_function_privilege('authenticated', …, 'EXECUTE') = false`
- `has_function_privilege('sandbox_exec',  …, 'EXECUTE') = false`
- `has_function_privilege('service_role',  …, 'EXECUTE') = true`
- `has_function_privilege('postgres',      …, 'EXECUTE') = true` (owner)

A migration é fail-closed por construção: reaplicação em qualquer
ambiente onde a role sandbox re-adquira `EXECUTE` fora deste script
provocará re-REVOKE + assert. Nenhum caminho da migration esconde
falha por `WHEN OTHERS`.

Observação: o ambiente gerenciado pode reaplicar `EXECUTE` a
`sandbox_exec` após o commit da migration por processos automatizados
externos ao repositório. A migration não pode impedir concessões
posteriores emitidas por outro superusuário — o que ela garante é o
estado final do próprio DDL committado. Auditorias externas devem
verificar o ACL contra o estado esperado ao invés de confiar apenas na
execução histórica da migration.

## 4. §19 — Stop condition triggered

A especificação da SCP-012.0.2.2 exige, para marcar a etapa
`Ready for External Audit`, que **todos os 40 cenários** da matriz
executem com sucesso (`0 failed`, `0 skipped`). Durante o reconnaissance
obrigatório desta execução, foi confirmado por consulta direta ao
`information_schema` que quatro cenários da matriz são **estruturalmente
inexecutáveis** contra o schema atual do PostgreSQL do projeto:

| # | Cenário | Motivo estrutural |
|---|--------|-------------------|
| 12 | value_int negativo (ou value_decimal negativo) | `tenant_entitlements_int_non_negative_chk CHECK (value_int >= 0)` **e** `tenant_entitlements_decimal_non_negative_chk CHECK (value_decimal >= 0)`; idem em `commercial_plan_entitlements`. INSERT com valor negativo falha antes de qualquer chamada à RPC. |
| 20 | `limit = 2147483648` | `tenant_entitlements.value_int` e `commercial_plan_entitlements.value_int` são `integer` (int4, máximo `2147483647`). A migration SCP-012.0.2.1 ampliou apenas o **contrato interno** da RPC (`v_limit bigint`), sem alterar o tipo das colunas. `value_decimal` é `numeric(14,2)` — precisão total 14, máximo ≈ `1e12` — insuficiente para acomodar o valor. |
| 21 | `limit = 9007199254740991` (MAX_SAFE_INTEGER) | Mesmo argumento: `int4` não comporta; `numeric(14,2)` tampouco. |
| 22 | `limit = 9007199254740992` → `not_evaluated` | Mesmo argumento: o valor não pode ser inserido no schema para provocar o comportamento `not_evaluated` esperado. |

Consulta de evidência:

```sql
SELECT column_name, data_type, numeric_precision, numeric_scale
  FROM information_schema.columns
 WHERE table_schema = 'public'
   AND table_name IN ('tenant_entitlements', 'commercial_plan_entitlements')
   AND column_name IN ('value_int', 'value_decimal')
 ORDER BY 1,2;
```

Resultado:

```
value_decimal | numeric | 14 | 2
value_decimal | numeric | 14 | 2
value_int     | integer | 32 | 0
value_int     | integer | 32 | 0
```

Constraints observadas:

```
tenant_entitlements_int_non_negative_chk        CHECK (value_int     IS NULL OR value_int     >= 0)
tenant_entitlements_decimal_non_negative_chk    CHECK (value_decimal IS NULL OR value_decimal >= 0)
commercial_plan_entitlements_int_non_negative_chk     CHECK (value_int     IS NULL OR value_int     >= 0)
commercial_plan_entitlements_decimal_non_negative_chk CHECK (value_decimal IS NULL OR value_decimal >= 0)
```

Portanto, **quatro cenários da matriz (12, 20, 21, 22) não podem ser
materializados** sem alterar o schema de `tenant_entitlements` e
`commercial_plan_entitlements` (widening dos tipos numéricos e/ou
remoção parcial dos CHECKs de não-negatividade). Essa mudança está fora
do escopo da SCP-012.0.2.2 (a etapa é declaradamente sem `mutation de
membership`, sem `enforcement` e sem alteração da autoridade comercial
materializada em SCP-012.0.2). Alterar o CHECK de não-negatividade e/ou
promover as colunas para `bigint`/`numeric` amplo é uma decisão
arquitetural própria — deve ser tratada em uma etapa dedicada
(candidato: `SCP-012.0.2.3 — Entitlement Numeric Column Widening`),
com sua própria análise de impacto, migração transacional e
comprovação de compatibilidade com o resto do domínio comercial.

Pelo próprio §19 da especificação:

> Se qualquer um ocorrer: […] qualquer cenário for ignorado; qualquer
> cenário falhar; […] então: não marcar SCP-012.0.2.2 Ready for
> External Audit; não atualizar o roadmap como concluído; não autorizar
> SCP-012.0.3; registrar a falha exata; manter a etapa Blocked.

Este documento cumpre esse mandato: registra a falha exata
(§4 acima), mantém a etapa em **Blocked**, e **não** autoriza
SCP-012.0.3.

## 5. Trabalho **não** realizado nesta execução

Explicitamente **não** foram materializados nesta execução, para não
introduzir artefatos parciais que criem falsa sensação de paridade:

- runner service-role de paridade SQL × TypeScript de 40 cenários
  (§5, §6, §13);
- lifecycle de usuários temporários via `supabase.auth.admin.createUser`
  no runner de paridade (§7);
- fixtures runId-prefixadas exclusivamente para o harness (§8);
- verificação pós-cleanup de resíduos por prefixo (§10);
- privilege verification real (anon / authenticated / service_role) via
  três clients (§14).

O harness `commercial-seat-sql-parity.spec.ts` e
`run-commercial-sql-parity-specs.ts` da SCP-012.0.2.1 permanecem no
repositório em seu estado anterior. Eles **não** representam o harness
service-role exigido pela SCP-012.0.2.2 e não são executados por esta
etapa. A rejeição da SCP-012.0.2.1 já foi registrada por auditoria
externa; este documento mantém 16.0.2.1 em `Blocked`.

## 8. Riscos e limitações reais

1. **Bloqueio estrutural** — o widening numérico das colunas
   `value_int` / `value_decimal` e a decisão sobre negativos são
   pré-requisitos para a paridade completa exigida pela SCP-012.0.2.2.
   Sem essa etapa, quatro cenários permanecem inexecutáveis.
2. **Sandbox re-grant** — o ACL do resolver pode ser reaplicado por
   processos externos ao repositório. A migration §15 garante o
   estado imediatamente após seu commit, não indefinidamente.
3. **Harness service-role ainda não materializado** — o comprovante
   executável dos 40 cenários exige uma etapa distinta, após a
   remoção do bloqueio estrutural, para não gerar suíte parcial.
4. **SCP-012.0.3 não autorizada** — a cadeia `Blocked` é preservada
   integralmente.
