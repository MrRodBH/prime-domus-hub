# SCP-012.0.2 — Transaction-Safe Commercial Authority Materialization & Atomic Runtime Cutover

## Status

Accepted

## 1. Escopo corrigido

Esta etapa materializa **apenas** a autoridade comercial read-only e o
cutover atômico do runtime de seat-limit para essa autoridade. O
boundary de membership pertence à **SCP-012.0.3** e o enforcement
decisão+mutation pertence à **SCP-012**.

Nesta execução:

- zero mutation em `public.tenant_members`;
- zero lock transacional;
- zero enforcement;
- zero payload de membership inventado;
- zero teste concorrente de mutation.

A ausência de mutation de membership no runtime **não** é divergência
— é precisamente o motivo pelo qual a SCP-012.0.3 existe.

## 4. Assinatura da RPC

```
public.resolve_commercial_seat_decision(
  _actor_user_id      uuid,
  _tenant_id          uuid,
  _tenant_origin      text,
  _requested_increment integer
) RETURNS jsonb
LANGUAGE plpgsql
STABLE
SECURITY DEFINER
SET search_path = public, pg_temp
```

Somente contexto server-attested trafega (identidade nunca vem do
payload público). A função é read-only; sem `INSERT/UPDATE/UPSERT/
DELETE`, sem `SELECT FOR UPDATE`, sem `pg_advisory_xact_lock`, sem
triggers.

## 5. Algoritmo SQL

Ordem canônica reproduzida (SCP-012.0.1 §algoritmo):

1. valida parâmetros (`_actor_user_id`, `_tenant_id`, `_tenant_origin
   ∈ {impersonation,selection,single-membership}`, `_requested_increment
   = 1`);
2. revalida `_actor_user_id` em `auth.users` e `_tenant_id` em
   `public.tenants`;
3. deriva `super_admin` de `public.user_roles`; exige coerência
   `super_admin ↔ impersonation` e `regular ↔ {selection,
   single-membership}`; usuário comum requer membership `active`;
4. `featureKey` fixado em `users.seats` (cataloged);
5. seleção determinística de subscription;
6. resolução de entitlement (tenant > plan) usando precedência
   `value_bool > value_int > value_decimal (inteiro em faixa) >
   value_text`; `effective_from`/`effective_until` respeitados na camada
   tenant;
7. billing status derivado das mesmas famílias de status usadas em
   `deriveBillingHealth`;
8. feature-gate:
   `billing_blocked > billing_attention_required > entitled/not_entitled
   por efetividade > billing_unknown > not_entitled/none`;
9. short-circuit negativo devolve DTO com `limit/used/remaining = null`
   e **não** consulta `tenant_members`;
10. limite ausente/inválido devolve `not_evaluated / none`;
11. `SELECT COUNT(*) ... WHERE membership_status IN ('active','invited')`;
12. `used + 1 <= limit ? entitled : limit_reached` com `remaining =
    max(limit - used, 0)` para `entitled` e `remaining = 0` para
    `limit_reached`.

## 9. Runtime cutover

`getCommercialSeatLimitDecision`
(`src/lib/api/commercial/commercial.functions.ts`) passou a:

- preservar o `inputValidator` (`normalizeCommercialSeatLimitInput`) e
  o middleware `requireTenant`;
- derivar `actorUserId = context.tenant.userId`,
  `tenantId = context.tenant.tenantId`,
  `tenantOrigin = context.tenant.origin` server-side;
- carregar `supabaseAdmin` dentro do handler (`await loadAdmin()`);
- invocar `admin.rpc("resolve_commercial_seat_decision", { ... })`;
- validar a resposta com `validateSeatDecisionResponse` (validator
  estrito interno);
- retornar o DTO canônico sanitizado.

O caminho antigo TypeScript foi removido do fluxo produtivo:
`resolveCommercialSeatLimitDecision`, `readCommercialSeatUsage`,
`defaultEvaluateCatalogGate` e `loadTenantCommercialContext` deixaram
de ser referenciados pelo seat-limit em produção. Os módulos
permanecem no repositório para uso por outras superfícies comerciais
e pelo harness de paridade das specs (SCP-012.0.2 §14) — sem chamada
pelo runtime público.

## 10. Ausência de dual authority

O `handler` da server function pública:

- não invoca `resolveCommercialSeatLimitDecision`;
- não invoca `readCommercialSeatUsage`;
- não invoca `decideCommercialFeature` para seat-limit;
- não invoca `decideCommercialSeatLimit`;
- não invoca `loadTenantCommercialContext` para seat-limit;
- não possui feature flag SQL vs TS;
- não possui `catch` que faça fallback TypeScript;
- em erro da RPC ou resposta inválida, falha deterministicamente
  (`throw new Error(...)`).

## 11. Validator de resposta

`validateSeatDecisionResponse` recusa:

- objetos que não sejam plain objects;
- campos desconhecidos (whitelist explícita);
- `tenantId` divergente do contexto;
- `featureKey` diferente de `users.seats`;
- `reason` fora do enum canônico;
- `source` fora do enum canônico;
- `limit/used/remaining` não-inteiros, negativos, NaN, Infinity ou
  decimais;
- `requestedIncrement` diferente do valor enviado (sempre 1);
- incoerência estrutural (`allowed=true` sem `reason=entitled`;
  `entitled` sem limit/used/remaining; `limit_reached` sem
  limit/used; `remaining ≠ 0` em `limit_reached`).

## 13. Testes executados

Contra PostgreSQL real via `psql`:

- Trusted Actor Context:
  - super admin + impersonation em tenant existente → retorna DTO
    canônico (`billing_unknown / none` para o tenant real amostrado,
    sem subscription);
  - super admin + `selection` → `ERROR: Super admin requires
    impersonation origin`;
  - super admin + `single-membership` → mesma rejeição;
  - `_tenant_origin = 'bogus'` → `ERROR: Invalid tenant origin`;
  - tenant inexistente → `ERROR: Tenant not found`;
  - `_requested_increment = 2` → `ERROR: Invalid requestedIncrement`.
- Privileges:
  - `has_function_privilege('anon', ..., 'EXECUTE') = false`;
  - `has_function_privilege('authenticated', ..., 'EXECUTE') = false`;
  - `has_function_privilege('service_role', ..., 'EXECUTE') = true`.

Runtime (estático):

- `bunx tsc --noEmit -p tsconfig.json` → exit 0;
- `git diff --check` → limpo.

Limitações declaradas honestamente:

- **Paridade quantitativa completa TypeScript × SQL** (todas as linhas
  da matriz canônica com fixtures ricos) requer refactor do harness
  atual em `src/integrations/supabase/__tests__/commercial-seat-limit.spec.ts`,
  que hoje exercita `resolveCommercialSeatLimitDecision` diretamente
  em memória. Um wrapper que invoque a RPC contra o banco real usando
  cada fixture ainda **não** foi materializado nesta etapa. Essa
  materialização deverá compor a próxima onda de trabalho
  (SCP-012.0.2.1 — Parity Harness Materialization) e é o único item
  reservado a auditoria crítica externa desta cadeia. Todas as
  branches foram inspecionadas manualmente e o SQL foi construído
  linha a linha a partir do TypeScript equivalente.

## 15. Riscos residuais

- **Paridade quantitativa formal**: coberta por inspeção estática e
  smoke tests contra o banco real, mas não por um harness
  automatizado exaustivo. Auditoria externa deve validar a fidelidade
  linha a linha (§13 limitação).
- **billing_events**: hoje não é consultado no resolver SQL, coerente
  com o fato de que `deriveBillingHealth` também deriva o status
  exclusivamente a partir de `tenant_subscriptions` + presença de
  `tenant_billing_provider_mappings`; `lastBillingEventAt` era campo
  informativo do DTO `TenantBillingHealth`, não entrada da decisão.
  Nenhum caminho de decisão consome `billing_events`.
- **Comprometimento de `service_role`**: mitigação fora do escopo
  aplicativo (SCP-012.0.1.3 §9).

## 16. Escopo explicitamente fora desta etapa
