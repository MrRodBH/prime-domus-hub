# SCP-012.0.1.3.1.1 — Non-Self-Referential Git Evidence Model, Accepted Status Finalization & SCP-012.0.2 Gate Cleanup

## Status

Accepted

## 1. Problema da evidência autorreferencial

Os documentos da SCP-012.0.1.3.1 e o relatório 112 continham
formulações que prometiam registrar, dentro do próprio arquivo
versionado, o HEAD final do commit que passaria a conter esse mesmo
arquivo. Trechos remanescentes:

- "HEAD final e a lista de commits desta etapa serão registrados no
  relatório 112 após a materialização automática";
- "HEAD final: registrado após materialização automática desta
  etapa";
- "Commits criados: registrados após materialização automática";
- "resultado registrado ao final desta execução (esperado exit 0)";
- "esperado limpo";
- "Working tree final — Registrado após materialização automática.
  Esperado: limpo".

Essas formulações descrevem como futuro/esperado um resultado já
concluído e persistem placeholders em um documento aprovado pela
auditoria externa.

## 2. Motivo técnico da impossibilidade

Um arquivo versionado não pode registrar o SHA do commit que o
contém. Ao inserir o SHA, um novo commit é criado com um novo hash e
o valor gravado deixa imediatamente de ser o HEAD final. Exigir tal
registro é uma regressão autorreferencial: o próprio ato de cumprir
a exigência a invalida.

## 3. Modelo Baseline × Reviewed Materialization Head × Evidence Patch

Adota-se, como regra permanente para evidência Git em etapas
documentais, a seguinte tripartição:

- **Baseline da etapa** — commit imediatamente anterior ao início da
  implementação; persistido no documento.
- **Reviewed Materialization Head** — commit que contém a
  implementação completa e já foi verificado pela auditoria crítica
  externa; persistido no documento porque existe antes do início do
  patch documental posterior.
- **Evidence Patch Head** — SHA final do patch documental que
  finaliza o status. Não é persistido no próprio patch, pois isso
  seria autorreferencial. É apresentado no relatório textual de
  execução e verificado diretamente pela auditoria externa no
  repositório.

## 12. Regra permanente para evidência Git futura

Etapas documentais futuras deverão:

- persistir o **Baseline da etapa** no próprio documento;
- persistir o **Reviewed Materialization Head** apenas quando esse
  commit já existir e tiver sido auditado antes do início do patch
  atual;
- **não** persistir o SHA do próprio commit que contém o arquivo
  (Evidence Patch Head), apresentando-o exclusivamente no relatório
  textual de execução para verificação externa direta no
  repositório;
- registrar resultados de validação (`tsc`, `git diff --check`,
  buscas por placeholders, escopo) apenas como resultados
  concluídos, nunca como "esperado" ou "a registrar".
