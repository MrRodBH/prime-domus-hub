# SCP-012.0.1 — Canonical Decision Contract, Atomic Cutover Sequencing & Roadmap Cleanup

## Status

Accepted

## 1. Objetivo

Corrigir as inconsistências levantadas pela auditoria externa da
SCP-012.0 sem introduzir qualquer implementação. A etapa:

- elimina duplicidades e entradas antigas do roadmap;
- alinha integralmente a matriz de paridade ao DTO canônico
  `CommercialLimitDecision`;
- corrige o comportamento documentado de `billing_attention_required`
  para short-circuit;
- elimina o planejamento de dual resolver e o período persistido de
  dual authority;
- reclassifica a estratégia arquitetural como SQL/RPC pura
  (Estratégia A), sem conexão Postgres direta (Estratégia C não
  selecionada);
- consolida o sequenciamento futuro com cutover atômico em uma única
  subetapa.

Etapa exclusivamente documental. Nenhum resolver SQL, migration, RPC,
mutation ou enforcement é criado.

## 2. Inconsistências confirmadas

Levantadas pela auditoria externa da SCP-012.0 e corrigidas aqui:

1. **Matriz de paridade** usava razões e sources inventados
   (`feature_not_cataloged`, `limit_unavailable`, `limit_invalid`,
   `within_limit`, `allowed` como reason). Nenhum desses valores
   pertence ao DTO `CommercialLimitDecision` aprovado (SCP-011.1 /
   SCP-011.2).
2. **`billing_attention_required`** aparecia como "avaliado
   normalmente", o que contradiz o runtime aprovado — a razão comercial
   negativa produz short-circuit antes da avaliação quantitativa.
3. **Estratégia recomendada** classificada como "A + C parcial",
   sugerindo conexão Postgres direta ou transaction callback
   multi-statement em TypeScript. A implementação viável é A pura
   (SQL/RPC).
4. **Sequenciamento futuro** previa uma subetapa que materializava o
   resolver SQL sem cutover do runtime, criando um período em que
   duas autoridades coexistiam.
5. **Roadmap** trazia risco documentado de coexistência de entradas
   antigas (`Transaction-Safe Commercial Resolver Materialization`,
   `transaction-safe commercial authority unavailable`) — removidas.

## 3. Contrato canônico

O único contrato aceito para a decisão comercial é:

```ts
type CommercialLimitDecision = {
  tenantId: string;
  featureKey: string;
  allowed: boolean;
  reason:
    | "entitled"
    | "not_entitled"
    | "limit_reached"
    | "billing_unknown"
    | "billing_attention_required"
    | "billing_blocked"
    | "not_evaluated";
  source: "tenant" | "plan" | "default" | "none";
  limit: number | null;
  used: number | null;
  requestedIncrement: number;
  remaining: number | null;
};
```

Valores **proibidos** no contrato, matrizes e planejamento (não são
razões nem sources válidos; nunca podem aparecer no runtime):

- `feature_not_cataloged` — valor proibido;
- `limit_unavailable` — valor proibido;
- `limit_invalid` — valor proibido;
- `within_limit` — valor proibido;
- `allowed` como `reason` — valor proibido;
- `tenant_entitlement` — valor proibido;
- `plan_entitlement` — valor proibido;
- `override` como `source` — valor proibido;
- `system` como `source` — valor proibido;
- `unknown` como `source` — valor proibido.

## 4. Matriz de paridade final

`source do CommercialFeatureDecision` denota a `source` já emitida pelo
helper `decideCommercialFeature`. `source do limite` denota o valor
emitido por `extractSeatLimit`.

| Cenário | allowed | reason | source | limit | used | remaining |
|---|---|---|---|---|---|---|
| §7.1 Catalog gate negativo | false | `not_evaluated` | `none` | null | null | null |
| §7.2 Feature `not_entitled` | false | `not_entitled` | source do CommercialFeatureDecision | null | null | null |
| §7.2 Feature `billing_unknown` | false | `billing_unknown` | source do CommercialFeatureDecision | null | null | null |
| §7.2 Feature `billing_attention_required` | false | `billing_attention_required` | source do CommercialFeatureDecision | null | null | null |
| §7.2 Feature `billing_blocked` | false | `billing_blocked` | source do CommercialFeatureDecision | null | null | null |
| §7.2 Feature `not_evaluated` | false | `not_evaluated` | source do CommercialFeatureDecision | null | null | null |
| §7.3 Feature positiva + limite ausente/inválido | false | `not_evaluated` | `none` | null | null | null |
| §7.4 Feature positiva + limite válido + uso indisponível | false | `not_evaluated` | source do limite | limite resolvido | null | null |
| §7.5 `used + requestedIncrement <= limit` | true | `entitled` | source do limite | limite resolvido | uso resolvido | `max(limit - used, 0)` |
| §7.6 `used + requestedIncrement > limit` | false | `limit_reached` | source do limite | limite resolvido | uso resolvido | 0 |
| §7.7 `limit = 0`, `used = 0`, `requestedIncrement = 1` | false | `limit_reached` | source do limite | 0 | 0 | 0 |

Regras invariantes:

- catalog gate negativo **nunca** carrega contexto comercial nem
  consulta `tenant_members`;
- **toda** razão comercial negativa produz short-circuit antes da
  avaliação quantitativa;
- feature positiva com limite ausente/inválido não consulta
  `tenant_members`;
- `remaining` só é numérico quando existem `limit` e `used`;
- precedência de source preservada: `tenant > plan > default > none`.

## 5. Tratamento de billing

Toda razão comercial negativa (`not_entitled`, `billing_unknown`,
`billing_attention_required`, `billing_blocked`, `not_evaluated`)
produz:

- `allowed = false`;
- `reason` = razão exata do `CommercialFeatureDecision`;
- `source` = source exata do `CommercialFeatureDecision`;
- `limit = null`, `used = null`, `remaining = null`;
- **nenhuma leitura de `tenant_members`**;
- nenhuma reinterpretação de billing.

Especificamente, `billing_attention_required` **não** dispara avaliação
quantitativa normal — qualquer documentação anterior nesse sentido está
revogada.

## 6. Estratégia arquitetural final

**Estratégia A — SQL/Postgres como autoridade comercial canônica
única**, exposta somente por primitive RPC server-side restrita.

- PostgREST / RPC permanece como superfície de chamada;
- a transação ocorre **integralmente dentro da função Postgres**;
- **não** existe transação multi-statement controlada pelo TypeScript;
- **não** existe pool ou conexão Postgres direta adicional;
- Estratégia C (conexão Postgres direta / transaction callback
  multi-statement em TypeScript) **não** é selecionada;
- Estratégia B (snapshot materializado) permanece rejeitada.

A futura mutation atômica invoca uma única RPC (`SECURITY DEFINER`)
que, no mesmo plano transacional Postgres, adquire o lock em
`public.tenants`, executa o resolver comercial SQL e aplica a mutation.

## 7. Autoridade única

Prova de autoridade única (sem dual resolver e sem período persistido
de dual authority em produção):

1. o runtime TS delega integralmente à RPC SQL no **mesmo** conjunto
   de mudanças que materializa o resolver (SCP-012.0.2, cutover
   atômico);
2. após o merge da SCP-012.0.2, o caminho decisório TypeScript
   independente é **removido ou desativado** — não permanece como
   runtime alternativo, fallback ou precedência comercial secundária;
3. o resolver TypeScript legado é utilizado **apenas como oracle
   temporário** dentro da suíte de paridade **durante a execução** da
   SCP-012.0.2 e não sobrevive ao cutover;
4. em produção, existe **uma única autoridade comercial runtime** a
   partir do merge da SCP-012.0.2;
5. helpers TS remanescentes servem apenas como contrato de tipos,
   normalização de input e projeção DTO whitelisted.

## 8. Sequenciamento final

1. SCP-012.0 — Impact Analysis.
2. SCP-012.0.1 — Canonical Decision Contract, Atomic Cutover
   Sequencing & Roadmap Cleanup (esta etapa).
3. SCP-012.0.2 — Transaction-Safe Commercial Authority
   Materialization & Atomic Runtime Cutover — conjunto único e
   auditável contendo: materialização do resolver SQL; suíte de
   paridade; delegação do runtime; remoção/desativação do caminho TS
   independente; prova de ausência de dual authority persistida.
4. SCP-012.0.3 — Membership Mutation Boundary Planning &
   Materialization (define primeiro quais operações são necessárias ao
   produto).
5. SCP-012 — Atomic Enforcement Integration.

Regra explícita: **não existe** subetapa que materialize o resolver
SQL sem cutover do runtime na mesma etapa.

## 9. Membership Mutation Boundary

O planejamento do boundary permanece, mas não presume fluxo de convite
por e-mail. Domínio atual confirmado:

- não existe tabela separada de invitations;
- `invited` é um status de `public.tenant_members`;
- `user_id` é obrigatório (FK NOT NULL para `auth.users`);
- não existe runtime de convite ou aceite.

A SCP-012.0.3 deverá primeiro definir quais operações são realmente
necessárias ao produto.

Candidatas (compatíveis com o domínio atual):

- `createMembership` para usuário existente;
- alterar status (dentro de `membership_status`);
- alterar role (dentro de `tenant_role`);
- suspender;
- revogar;
- reativar.

Dependentes de análise adicional:

- invite por e-mail;
- aceite de convite;
- delete físico;
- criação de `auth user`.

Input permitido: `targetUserId`, `targetStatus`, `targetRole`.
Input proibido: `tenantId`, `limit`, `used`, `remaining`, `source`,
`billingStatus`, `seatDelta`, `featureKey`, qualquer
`CommercialLimitDecision`/`CommercialFeatureDecision`.
