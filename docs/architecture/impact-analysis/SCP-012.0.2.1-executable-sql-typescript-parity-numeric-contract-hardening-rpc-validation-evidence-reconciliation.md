## 7. Contrato numérico corrigido

O resolver usa exclusivamente `bigint`/`numeric` em `v_limit`, `v_used`,
`COUNT(*)` e comparações aritméticas. Constantes:

- `MAX_SAFE_INTEGER = 9007199254740991` verificado no PostgreSQL como
  `numeric`;
- valores fracionários rejeitados (`v_ent_value_decimal = floor(...)`);
- valores negativos rejeitados;
- casts `::bigint` só ocorrem após validação de range.

Zero uso de `integer/int4` em variáveis de limite/uso.

## 8. Determinismo de subscriptions

SQL e TypeScript agora usam a mesma ordenação canônica:

1. prioridade do status (CASE fixo);
2. `started_at DESC NULLS LAST`;
3. `id ASC` (tie-breaker final).

`pickActiveSubscription` em `commercial.functions.ts` foi atualizado para
incluir o passo (3).

## 9. Determinismo de entitlements

Após confirmar as constraints `UNIQUE(tenant_id, entitlement_key)` e
`UNIQUE(plan_id, entitlement_key)`, o resolver SQL confia nessa unicidade
e remove `ORDER BY effective_from DESC, id ASC` para tenant entitlements
(era supérfluo). No TypeScript, `deriveEntitlementSnapshot` já usa
`Map.set(entitlement_key, …)`, também determinístico.

## 10. Source `default`

**Decisão**: `default` permanece token reservado no enum DTO
(`CommercialLimitDecisionSource`) para forward-compatibility. Nenhum
produtor persistido canônico existe: nem `commercial_entitlement_definitions`,
nem `commercial_plan_entitlements`, nem `tenant_entitlements`, nem
`deriveEntitlementSnapshot` produzem `source='default'` em runtime.
Precedência realmente materializada: **tenant > plan > none**.

Documentos afetados (correção mínima):

- `SCP-012.0.1` e este documento registram explicitamente:
  precedência runtime = `tenant > plan > none`; `default` é reservado
  e inalcançável até que um produtor persistido seja definido em etapa
  futura dedicada.

## 12. Validator runtime

Extraído para `src/lib/api/commercial/seat-limit-rpc-contract.ts`
(módulo puro, sem imports supabase/server/React/rotas). Exporta
`validateSeatDecisionResponse`.

Cobertura semântica endurecida:

- rejeita `allowed=true + reason != entitled`;
- rejeita `entitled + source=none`;
- rejeita `entitled` com `used + increment > limit`;
- rejeita `entitled` com `remaining != max(limit - used, 0)`;
- rejeita `limit_reached` com `used + increment <= limit`;
- rejeita `limit_reached` com `remaining != 0`;
- rejeita `not_entitled|billing_*` com quaisquer campos quantitativos
  não-nulos;
- rejeita `billing_unknown` com `source != none`;
- aceita `not_evaluated` apenas nas duas formas canônicas (§13.12 do
  spec).

Suíte unitária `commercial-seat-rpc-contract.spec.ts` cobre 32 casos
(passed).

## 13. Harness executável de paridade

- `src/integrations/supabase/__tests__/commercial-seat-sql-parity.spec.ts`
- runner: `run-commercial-sql-parity-specs.ts`
- executa `SELECT public.resolve_commercial_seat_decision(...)` via `psql`
  contra o Supabase gerenciado;
- oracle TypeScript deriva o DTO a partir da mesma fixture in-memory;
- comparação por deep-equality (JSON canonicalizado).

Escopo real coberto:

- Cenários 1-11 e 15-17 (billing paths, entitlement bool/int/decimal,
  MAX_SAFE_INTEGER e boundary de int4, tie-breaker de subscription):
  fixtures materializáveis pela role `sandbox_exec` no ambiente atual.
- Cenários 12-14 (memberships active/invited/suspended/revoked):
  **NÃO EXECUTÁVEIS** pelo harness atual — o role `sandbox_exec` recebe
  `INSERT` em `tenant_members` mas RLS bloqueia via
  `permission denied for table tenant_members` durante `INSERT` sem
  contexto autenticado. Documentado como limitação a ser tratada em
  ambiente de teste isolado no próximo ciclo (§21 riscos residuais).

O runner encerra com `exit 1` em qualquer discrepância; o TypeScript
oracle percorre a mesma máquina de decisão canônica
(`decideCommercialFeature` + `extractSeatLimit` + `decideCommercialSeatLimit`)
já validada por 55 casos unitários (`commercial-seat-limit.spec`).

## 14. Fixtures isoladas

Cada cenário:

- gera UUIDs frescos para tenant, plan, subscription;
- respeita constraints reais (`tenants.status = 'ativo'`, slug regex,
  plan code regex, tenant_role enum);
- é removido em bloco `finally` na ordem inversa das FK.

Reutiliza apenas `auth.users` já existentes (super_admin + regulares),
sem criar/alterar usuários. Nenhum tenant produtivo é tocado.

## 15. Runtime cutover preservado

`getCommercialSeatLimitDecision`:

- mantém `requireTenant` + `normalizeCommercialSeatLimitInput`;
- deriva `actorUserId`/`tenantId`/`tenantOrigin` do context;
- carrega `supabaseAdmin` no handler;
- invoca `admin.rpc("resolve_commercial_seat_decision", …)`;
- valida com `validateSeatDecisionResponse` (agora importado do módulo puro);
- não possui try/catch de fallback, feature flag SQL/TS, nem chamada aos
  helpers TS (`resolveCommercialSeatLimitDecision`, `readCommercialSeatUsage`)
  no caminho produtivo.

## 16. Route tree

`src/routeTree.gen.ts` verificado: reflete o mapeamento de rotas atual

## 17. Confirmações negativas

- zero mutation em `tenant_members` (grep sem `INSERT|UPDATE|DELETE` do
  runtime; harness insere em fixtures descartáveis + `finally` cleanup);
- zero lock (`pg_advisory_xact_lock`, `FOR UPDATE` ausentes);
- zero enforcement (função `STABLE`, sem side-effects);
- zero fallback TypeScript no runtime público;
- zero role adicional com EXECUTE além de owner + service_role;
- zero secret exposto no diff;
- zero frontend alterado;
- zero SCP-012.0.3 iniciada.

## 18. Escopo NÃO implementado

- SCP-012.0.3 (Membership Mutation Boundary);
- SCP-012 (Atomic Enforcement);
- cobertura harness dos cenários 12-14 (membership fixtures) —
  bloqueada por RLS `tenant_members` no role de teste;
- cenários 25-30 (actor rejections) automatizados via harness — cobertos
  por smoke tests SQL manuais na SCP-012.0.2, não re-executados aqui;
- refinamento do sistema de fixtures via Supabase Admin API para criar
  auth.users de teste (fora do escopo desta correção).

## 19. Riscos residuais declarados

- **RLS em `tenant_members`**: o harness não consegue popular
  membership fixtures via `sandbox_exec` (permission denied), o que
  restringe a matriz executável aos cenários sem contagem de seats.
  A auditoria externa deve decidir se essa lacuna é aceitável nesta
  fase ou requer harness em role privilegiado.
- **Statuses `grace`/`unpaid` inalcançáveis**: o schema não os aceita;
  o código mantém o mapeamento por forward-compat mas os ramos são
  código morto até que o CHECK constraint seja ampliado.
- **`source = 'default'` inalcançável**: mantido no enum apenas por
  compatibilidade, sem produtor persistido.
- **Harness dependente de `psql`**: exige `PGHOST` etc.; não roda em
  CI que só tenha acesso via cliente Supabase JS.
